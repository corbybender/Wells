# Spec: Token Optimization Layer for Coding Agent Harness

## Objective

Build a token optimization layer into the LangChain/OpenHands-based coding agent harness to reduce total token usage without reducing task quality, correctness, or user experience.

The system should optimize token usage by controlling what context is sent to the model, summarizing stale state, compressing tool outputs, retrieving only relevant code, routing tasks to cheaper models when appropriate, and measuring token spend by category.

This is not primarily a “rewrite the user’s question” feature. User prompts are usually small. The major token cost comes from system prompts, chat history, repository context, tool outputs, logs, diffs, and repeated orchestration state.

## Goals

1. Reduce total input tokens per agent step.
2. Reduce repeated context across steps.
3. Preserve all information required for correct coding behavior.
4. Avoid lossy compression of critical instructions.
5. Provide observability into token usage by category.
6. Support arbitrary downstream AI models.
7. Keep the optimization layer model-agnostic where possible.
8. Allow configurable budgets per model, task type, and phase.

## Non-Goals

1. Do not blindly compress every user message.
2. Do not remove safety, system, or developer instructions.
3. Do not summarize source code when exact code is required for editing.
4. Do not hide context-reduction decisions from logs.
5. Do not rely on one vendor-specific tokenization API only.

## Core Architecture

Introduce a `ContextManager` component responsible for constructing the model input for every agent step.

```text
User Request
   ↓
Intent / Task Classifier
   ↓
ContextManager
   ├── System Prompt Manager
   ├── Conversation Trimmer
   ├── Task State Summarizer
   ├── Code Retriever
   ├── Tool Output Compressor
   ├── Token Budget Allocator
   ├── Model Router
   └── Token Usage Logger
   ↓
Model Invocation
```

## Component Requirements

### 1. Token Budget Allocator

Create a budget allocator that assigns token limits by category.

Example categories:

```text
system_prompt
developer_instructions
user_request
recent_conversation
task_state_summary
retrieved_code
tool_outputs
test_logs
diffs
model_response
```

The allocator should support configuration like:

```yaml
token_budgets:
  default:
    max_input_tokens: 24000
    reserved_output_tokens: 4000
    categories:
      system_prompt: 4000
      recent_conversation: 3000
      task_state_summary: 1500
      retrieved_code: 10000
      tool_outputs: 4000
      diffs: 1500
```

Acceptance criteria:

* Every model call is assembled under a known token budget.
* If the budget is exceeded, the system trims or summarizes lower-priority context first.
* System/developer instructions are never dropped.
* The budget decision is logged.

### 2. Conversation Trimming

Implement conversation trimming for old chat messages.

Keep:

* the original user request
* the most recent turns
* explicit user constraints
* unresolved questions
* task-critical decisions

Summarize or drop:

* stale planning chatter
* repeated assistant explanations
* obsolete failed attempts
* verbose intermediate logs already captured elsewhere

Acceptance criteria:

* Recent turns remain verbatim.
* Older turns are replaced by a structured task summary.
* No user-stated requirement is dropped without being preserved in summary form.

### 3. Task State Summary

Maintain a durable rolling task summary.

Use this schema:

```markdown
## Task State Summary

### Goal
...

### User Constraints
...

### Current Plan
...

### Files Inspected
...

### Files Modified
...

### Important Findings
...

### Commands Run
...

### Current Error / Failing Test
...

### Open Questions
...

### Next Action
...
```

Update the summary after each significant agent step.

Acceptance criteria:

* The summary is short enough to fit its assigned budget.
* It preserves durable facts and decisions.
* It avoids verbose reasoning.
* It can replace older conversation history safely.

### 4. Code Retrieval Instead of Full Repo Injection

The harness should not send the full repository by default.

Implement code retrieval using one or more of:

* file path matching
* symbol search
* embedding search
* ripgrep-style keyword search
* import graph traversal
* test-to-source mapping
* recent error stack trace extraction

Retrieve context units such as:

```text
file path
class/function body
nearby imports
callers/callees
related tests
configuration files
current diff
```

Acceptance criteria:

* Only top relevant files/chunks are sent.
* Exact source code is included when edits are required.
* Large files are chunked by symbol or logical section.
* Retrieval decisions are logged with relevance reasons.

### 5. Tool Output Compressor

Compress shell, test, build, and lint outputs before sending them to the model.

Keep:

```text
command
exit code
short status
relevant error lines
stack trace root cause
file/line references
last 100-200 useful lines
```

Remove:

```text
progress bars
installation noise
repeated stack frames
successful verbose logs
duplicate warnings
ANSI escape codes
```

Acceptance criteria:

* Raw logs are stored outside the model context.
* Compressed logs are sent to the model.
* The model can request raw logs if needed.
* Test failures preserve enough context to debug.

### 6. Prompt Caching Support

Keep stable prompt prefixes byte-for-byte identical whenever possible.

Stable prefix should include:

```text
system prompt
developer instructions
tool descriptions
agent operating rules
response format rules
```

Variable context should come after the stable prefix.

Acceptance criteria:

* Prompt prefix ordering is deterministic.
* Tool schema ordering is deterministic.
* Stable instructions are not regenerated differently each turn.
* Cache-hit-compatible prompt construction is logged where supported.

### 7. Model Router

Add model routing by task type.

Use cheaper/faster models for:

```text
intent classification
query rewriting
log summarization
file ranking
diff summarization
test output compression
simple validation
```

Use stronger models for:

```text
architecture changes
multi-file edits
ambiguous bug fixing
complex reasoning
security-sensitive changes
final patch synthesis
```

Acceptance criteria:

* Routing rules are configurable.
* The selected model and reason are logged.
* The system can fall back to a stronger model when confidence is low.

### 8. Query Rewriting

Implement query rewriting only where it helps retrieval or planning.

Good uses:

```text
turn vague user request into search queries
extract file names, symbols, errors, technologies
create concise retrieval queries
normalize issue descriptions
```

Bad uses:

```text
compressing already-short user prompts
dropping nuance from user requirements
replacing the original user request
```

Acceptance criteria:

* Original user request is always retained.
* Rewritten query is used as supplemental metadata.
* Rewriting is skipped when it would cost more than it saves.

### 9. Token Usage Observability

Log token usage by category and step.

Example event:

```json
{
  "step_id": "agent-step-12",
  "model": "example-model",
  "task_type": "debug_test_failure",
  "tokens": {
    "input_total": 18420,
    "system_prompt": 3120,
    "conversation": 1900,
    "task_summary": 850,
    "retrieved_code": 8200,
    "tool_outputs": 3750,
    "diffs": 600,
    "output_total": 2100
  },
  "context_actions": [
    "trimmed_conversation",
    "compressed_test_log",
    "retrieved_6_code_chunks"
  ]
}
```

Acceptance criteria:

* Token usage is attributable.
* The system can report top token-cost categories.
* Logs support comparing before/after optimization.

## Context Assembly Priority

When context exceeds budget, trim in this order:

1. verbose tool output
2. stale assistant messages
3. duplicate retrieved chunks
4. old conversation turns
5. low-relevance code chunks
6. historical summaries
7. diffs not relevant to current step

Never trim:

1. system instructions
2. developer instructions
3. current user request
4. explicit user constraints
5. exact code currently being edited
6. active error message or failing test details

## Suggested Interfaces

### ContextManager

```python
class ContextManager:
    def build_context(
        self,
        task: AgentTask,
        conversation: list[Message],
        repo_state: RepoState,
        tool_history: list[ToolResult],
        budget: TokenBudget,
    ) -> ModelContext:
        ...
```

### TokenBudget

```python
class TokenBudget:
    max_input_tokens: int
    reserved_output_tokens: int
    category_limits: dict[str, int]
```

### ContextReport

```python
class ContextReport:
    input_tokens_estimated: int
    category_tokens: dict[str, int]
    trimmed_items: list[str]
    summarized_items: list[str]
    retrieved_items: list[str]
    warnings: list[str]
```

## Evaluation Plan

Measure before and after:

```text
average input tokens per step
average output tokens per step
total tokens per task
successful task completion rate
number of model calls per task
number of failed tool calls
number of times raw logs were requested
latency per task
cost per task
```

Create benchmark tasks:

```text
small single-file bug fix
multi-file feature implementation
test failure debugging
dependency/configuration issue
large log investigation
repo onboarding question
```

Success criteria:

* At least 30% reduction in average input tokens.
* No meaningful drop in task success rate.
* No increase in average repair attempts.
* Tool-output-heavy tasks should show the largest savings.

## Implementation Phases

### Phase 1: Observability

Add token accounting by category.

Deliverables:

* token estimator
* context category labels
* per-step token report
* before/after dashboard or logs

### Phase 2: Tool Output Compression

Add structured compression for logs, tests, and shell output.

Deliverables:

* ANSI cleanup
* duplicate-line removal
* stack trace extraction
* relevant error detection
* raw log fallback

### Phase 3: Conversation Trimming and Task Summary

Add rolling task summaries and history pruning.

Deliverables:

* task summary schema
* summary update logic
* recent-message retention policy
* explicit user constraint preservation

### Phase 4: Code Retrieval

Replace full repo injection with targeted retrieval.

Deliverables:

* file/symbol search
* stack-trace-to-file retrieval
* test-to-source retrieval
* relevance scoring
* chunk budgeting

### Phase 5: Model Routing and Prompt Caching

Add deterministic prompt construction and model routing.

Deliverables:

* stable prompt prefix
* deterministic tool ordering
* routing config
* fallback strategy

## Risks

### Risk: Over-compression loses critical information

Mitigation:

* preserve original user request
* preserve exact active error
* keep exact code for edited files
* allow raw context fallback

### Risk: Retrieval misses relevant files

Mitigation:

* combine keyword, symbol, import graph, and embedding retrieval
* include current diff
* expand context when model reports missing information

### Risk: Summaries become inaccurate

Mitigation:

* use structured summaries
* update only from observed facts
* retain recent verbatim messages
* periodically regenerate summary from canonical event log

### Risk: Token estimates vary by model

Mitigation:

* use model-specific tokenizers when available
* otherwise use conservative estimates
* leave safety margin in token budget

## Final Instruction to Coding Agent

Implement the token optimization layer as a first-class orchestration subsystem. Do not treat it as a prompt-only change. The main value comes from context selection, state condensation, retrieval, compressed tool outputs, prompt caching, routing, and observability.
