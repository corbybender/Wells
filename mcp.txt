You are working in my existing repository.

I have already built a coding-agent harness as a package inside this repo. The harness should remain the core orchestration runtime. I want you to add an MCP server interface around the existing harness without rewriting the harness itself.

## Goal

Expose the existing coding-agent harness as a Model Context Protocol server so external agent clients such as Claude Code, OpenCode, Codex-style CLIs, Gemini CLI, or other MCP-compatible tools can call into the harness.

The MCP server should be an additional interface layer, not a replacement for the current CLI/package architecture.

## Desired Architecture

```text
External MCP Client
    ↓
MCP Server Interface
    ↓
Existing Harness Core
    ↓
Repo tools / context manager / retrieval / execution / model provider
```

Do not duplicate orchestration logic in the MCP layer. The MCP layer should be thin and should delegate to the existing harness services.

## Requirements

1. Inspect the existing package structure and identify the current harness entry points.
2. Preserve existing CLI behavior.
3. Add an MCP server module under the existing package.
4. Expose harness capabilities as MCP tools.
5. Reuse the existing context manager, retrieval system, task runner, tool executor, and model-provider abstraction.
6. Add configuration so the MCP server can be started from the command line.
7. Add tests for the MCP server tool registration and basic tool invocation.
8. Add documentation showing how to run the MCP server and connect an MCP-compatible client.

## Suggested MCP Tools

Expose tools similar to these, adjusted to match the actual harness APIs:

```text
run_agent_task
get_task_state
search_repo
retrieve_context
run_command
run_tests
summarize_logs
create_patch
review_diff
```

Each tool should have a clear JSON schema, typed inputs, typed outputs, and error handling.

## Implementation Guidelines

* Keep the MCP layer thin.
* Do not hardcode one model provider.
* Do not bypass the existing token optimization/context-management layer.
* Do not send the full repo context by default.
* Preserve existing logging and observability.
* Make tool outputs compact and structured.
* Ensure long-running tasks return useful status or task IDs if the current harness supports that.
* Avoid adding large new dependencies unless necessary.
* Follow the repository’s existing style, typing, and test conventions.

## CLI Requirement

Add a command similar to one of these, depending on the project’s existing CLI style:

```bash
harness mcp serve
```

or:

```bash
python -m harness.mcp_server
```

The server should support stdio transport at minimum because most MCP clients support it. HTTP/SSE transport can be optional if the MCP library makes it straightforward.

## Deliverables

1. MCP server implementation.
2. Tool wrappers that delegate to existing harness functions.
3. CLI command to start the server.
4. Tests.
5. README/docs section with usage examples.
6. Brief implementation summary explaining what changed and how external clients should use it.

## Acceptance Criteria

* Existing harness CLI still works.
* MCP server starts successfully.
* MCP client can discover the exposed tools.
* At least one tool can invoke the existing harness on a simple task.
* Tool responses are structured and compact.
* No orchestration logic is duplicated in the MCP server layer.
* Existing context/token optimization path is preserved.
