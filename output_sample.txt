ARCHITECTURE PROPOSAL
======================================================================
Based on the provided development plan and operating principles, here is a concrete architectural design for the Admin File Explorer.

Because Phase 1 (Discovery) has not been completed yet, this design relies on **Hypothesized** assumptions (A1-A4) and is structured to be highly modular so it can be integrated cleanly regardless of the exact existing framework.

---

### 1. Components / Modules and their Responsibilities

To adhere to the "Simplicity First" and "Surgical Changes" principles, this will be implemented using vanilla JavaScript with clear separation of concerns.

*   **Backend File Service (`api/admin-service.js`)**
    *   *Responsibility:* Serve the raw file data for a given site.
    *   *Change required:* Ensure it returns an array of relative path strings (e.g., `["css/style.css", "about/index.html"]`). If it currently returns a nested object, no change is needed.
*   **Tree Builder Utility (`admin/js/file-tree.utils.js` - *New File*)**
    *   *Responsibility:* Pure, deterministic data transformation.
    *   *Function:* `buildTree(flatPaths)` takes an array of strings and outputs a nested `TreeNode` schema. It will handle path normalization (converting `\` to `/`) and enforce sorting (Folders first, then Files, A-Z).
*   **File Explorer View (`admin/js/file-explorer.ui.js` - *New File*)**
    *   *Responsibility:* Render the `TreeNode` data into the DOM and manage local UI state.
    *   *Functions:*
        *   `renderTree(treeData, container)`: Generates the nested HTML structure.
        *   `toggleFolder(nodeId)`: Toggles expand/collapse classes.
        *   `selectFile(nodeId)`: Highlights the file and dispatches an event enabling the "Edit Now" button.
*   **Admin Controller (`admin/index.html` / inline script)**
    *   *Responsibility:* Glue layer. Listens for site selection, calls the Backend File Service, passes data to the Tree Builder, hands the result to the File Explorer View, and wires the "Edit Now" button to the existing editor function.

### 2. Key Data Models / Schemas

**A. API Response Payload (Hypothesized)**
```json
[
  "index.html",
  "about/team.html",
  "css/styles.css"
]
```

**B. Internal Tree Data Structure (Observed via design)**
The `buildTree` utility will output an array of nested objects:
```typescript
interface TreeNode {
  id: string;          // e.g., "about/team.html" (the full relative path)
  name: string;        // e.g., "team.html"
  path: string;        // e.g., "about/team.html"
  type: 'folder' | 'file';
  children: TreeNode[]; // Empty array for files
}
```

**C. DOM Structure Schema**
A standard accessible `<ul>` / `<li>` structure.
```html
<div id="file-explorer-container">
  <ul class="tree-root">
    <li class="tree-folder is-expanded">
      <div class="tree-node-header"><span class="caret">▾</span> 📁 about</div>
      <ul class="tree-children">
        <li class="tree-file is-selected" data-path="about/team.html">
          <span class="caret-placeholder"></span> 📄 team.html
        </li>
      </ul>
    </li>
  </ul>
</div>
<button id="edit-now-btn" disabled>Edit Now</button>
```

### 3. External Dependencies and Libraries

*   **Vanilla JavaScript:** No external JS frameworks (React, Vue, jQuery) will be introduced. This prevents bloat and adheres to "Surgical Changes."
*   **CSS:** Styling will be added to the existing admin CSS file (or a new `file-explorer.css` file linked only on this page).
*   **Icons:** Inline SVGs or standard Unicode emoji (📁, 📄, ▾, ▸) to match existing project style without adding heavy icon libraries like FontAwesome (unless one is already present).

### 4. Risk Areas / Trade-offs

*   **Risk: DOM Bloat on Large Sites.**
    *   *Context:* If a site has 5,000 files, rendering the entire tree eagerly will freeze the browser tab.
    *   *Trade-off:* Build the entire tree eagerly for now (Simplicity First). If Phase 1 or testing reveals massive file counts, we will need to pivot to "Lazy Loading" (only fetching/rendering sub-folders when the parent is expanded), which significantly increases complexity.
*   **Risk: Unsaved Changes in Editor.**
    *   *Context:* If a user clicks "Edit Now", does the existing editor prompt them to save unsaved work?
    *   *Mitigation:* The "Edit Now" button must simply call the existing `loadFileIntoEditor(path)` function. We must NOT build our own unsaved-changes guard; we rely on the existing editor's built-in protections.
*   **Risk: Path Separator Inconsistency.**
    *   *Context:* Windows backslashes vs Unix forward slashes.
    *   *Mitigation:* The `buildTree` utility will sanitize all inputs immediately using `path.replace(/\\/g, '/')` before splitting strings into tree nodes.
*   **Risk: Context Loss during State Changes.**
    *   *Context:* If the user collapses a folder, selects a different site, and comes back, is the tree state preserved?
    *   *Trade-off:* State will be kept strictly in memory for the active session. Switching sites will likely destroy the current tree state to save memory, requiring a re-fetch if they switch back (standard SPA behavior).
